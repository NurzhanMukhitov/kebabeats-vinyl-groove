## Текущий статус

| Фаза | Состояние |
|------|-----------|
| **A** — инфра и данные | **Готово:** Redis в Production, Vercel Analytics с `@vercel/analytics`, API `/api/play-counts` проверен; PWA не перехватывает `/api/*` (`navigateFallbackDenylist` в `vite.config.ts`). |
| **B** — расширенная статистика | **Не делаем:** достаточно визитов + агрегированных play counts по трекам; отдельная radio-статистика не нужна. |

## Цели

1. **Понимать использование продукта** — не только визиты (Vercel Analytics), но и осмысленные действия там, где это оправдано.
2. **Стабильный опыт** — радио и миксы без сюрпризов на мобильных и в слабой сети.
3. **Дешёвое сопровождение** — тесты, актуальная документация, разумный размер зависимостей.

## Фаза A — Инфраструктура и доверие к данным

| # | Действие | Зачем |
|---|----------|--------|
| A1 | Проверить в Vercel **Settings → Environment Variables**: Redis/KV для **Production** (и при необходимости Preview) совпадают с `api/_lib/redis.ts` | Счётчики миксов работают в проде, а не только локальный fallback |
| A2 | Убедиться, что **Vercel Web Analytics** получает события после деплоя с `@vercel/analytics` (зайти на прод, подождать ~час) | Валидация включённой аналитики |
| A3 | Синхронизировать **PROJECT.md**: версия Vite и упоминание Vercel Analytics | Меньше путаницы у разработчиков |

**Критерий готовности фазы:** в проде счётчики не «пустые» при наличии Redis; Analytics показывает реальные визиты.

## Фаза B — Статистика продукта (по желанию)

| # | Действие | Зачем |
|---|----------|--------|
| B1 | Зафиксировать **решение**: нужны ли **custom events** (Vercel Pro / другой провайдер) или достаточно визитов + Redis | Не строить лишнее |
| B2a | Если нужны события: спроектировать минимальный набор (например: `mix_play`, `radio_station`, `tab`) и место вызова (`Index`, хуки) | Понимание «что слушают» на уровне событий |
| B2b | Альтернатива: **агрегаты в Redis** — отдельные счётчики по `stationId` / дням через новые эндпоинты | Своя аналитика без сторонних SDK, только сервер |
| B3 | При расширении трекинга за пределы Vercel — оценить **privacy / cookie** для аудитории в EU | Юридическая гигиена |

**Критерий:** есть осознанный минимум данных, который отвечает на вопросы продукта, без дублирования трёх систем.

## Фаза C — Надёжность и UX

| # | Действие | Зачем |
|---|----------|--------|
| C1 | **Радио при обрыве сети:** сообщение пользователю + повтор при восстановлении (или явная кнопка) | Меньше «молчаливого» зависания |
| C2 | Ревизия **мини-плеера / safe-area** на новых iPhone после изменений | Стабильный вид на всех вырезах |
| C3 | При необходимости — тот же `handleMixTogglePlay`-подход для **других** точек входа play/pause (если появятся) | Единая политика «радио vs микс» |

## Фаза D — Качество и сопровождение

| # | Действие | Зачем |
|---|----------|--------|
| D1 | **Playwright:** 1–2 smoke-сценария (открыть `/`, вкладка, базовый UI) | Регрессии после деплоев |
